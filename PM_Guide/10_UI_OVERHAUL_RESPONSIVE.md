# UI 全面翻新 + 響應式 — 分析與計畫

> 起因：使用者指出（1）美感缺乏分析、(2) 完全沒有響應式、手機跑版、桌面是細欄飄空白。
> 本文件先「分析為什麼」，再定落地計畫。對應設計章程支柱 6（含 RWD）與整體視覺品質。
> 日期：2026-06-10

---

## 一、客觀診斷（程式碼證據，非臆測）

- **7 個頁面/元件硬編 `max-w-md`**（~448px 置中欄）：MyDayPage / TodoList / CalendarPage /
  TimelinePage / Archive / AddTodoDialog / BottomNav。
- **app 自有頁面幾乎零響應式斷點**：`sm:/md:/lg:` 幾乎只出現在 shadcn `ui/` 函式庫元件，
  我們自己寫的頁面沒有任何斷點。
- **結論**：不是「某頁壞掉」，是**根本沒有響應式系統** —— 所有螢幕都套同一條窄欄 + 底部導航。
  - 桌面（≥768px）：448px 細欄飄在大片空白漸層，像沒做完。
  - 手機：底部 5 格導航、固定字級、對話框在窄螢幕易擠壓/溢出。

## 二、研究與「為什麼好看／好用」的分析

業界（Material 3、Apple HIG、主流任務 app）的響應式導航是一條**自適應光譜**：

| 尺寸 | 導航形態 | 為什麼 |
|---|---|---|
| 手機 (<640px) | 底部 Tab Bar（3–5 項） | 拇指最易觸及的螢幕底緣；單手操作 |
| 平板 (≥768px) | Navigation Rail（窄垂直欄，圖標+標籤） | 螢幕變寬，底部過寬難觸及；側欄善用水平空間 |
| 桌面 (≥1024px) | 完整 Sidebar | 寬螢幕適合常駐側欄 + 寬內容區，資訊密度高 |

**為什麼這比「到處 max-w-md」好看**：
1. **善用空間**：桌面不再是空白漸層配細欄，而是側欄 + 寬內容，視覺平衡、像完整產品。
2. **觸及性**：手機底部導航符合拇指熱區；桌面側欄符合滑鼠/鍵盤習慣。
3. **一致的選中態**：用顏色/權重明確標示當前位置（研究強調，使用者不該用猜的）。
4. **觸控目標 ≥44×44pt**：間距足夠避免誤觸。

## 三、落地計畫（分階段，每階段一 PR、可視覺驗證）

### 階段 A：響應式 AppShell（結構，最高優先）
- 新建 `AppShell` 佈局元件：
  - `<md`：維持底部 BottomNav（既有）。
  - `≥md`：左側 Sidebar/Rail 導航，隱藏底部導航。
  - 內容容器：手機 `max-w-md`，平板/桌面放寬（`md:max-w-2xl lg:max-w-3xl` + 居中或側欄並排）。
- 用 React Router 的 layout route 包住所有頁面，移除各頁各自的 `max-w-md` 與 `<BottomNav/>`，
  收斂到單一 shell（順帶解決重複）。

### 階段 B：內容區響應式
- 各頁列表/卡片在寬螢幕用多欄 grid（例如 My Day 雙欄：今日 / 逾期+聚焦）。
- 字級、間距引入斷點階梯（避免手機過大、桌面過小）。

### 階段 C：視覺精緻化（需使用者視覺 review）
- 間距與字體階梯系統化（type scale）、卡片層次、焦點環、hover/active 一致化。
- confetti 改海洋色票；glass 卡片對比度微調。
- 大改色彩/風格的方向會做 A/B 截圖供使用者選，不擅自定案。

## 四、驗證方式

- 每階段用 vite + Preview 在 **mobile(375) / tablet(768) / desktop(1280)** 三寬度截圖比對。
- `vite build` 必過。
- 注意：Preview 的 `npm run dev` 在此環境無法綁定 port，改用 Bash `npx vite` + 瀏覽器導向
  （見記憶 preview-dev-server-quirk）。

## 五、tech lead 取捨聲明

- **響應式（階段 A/B）是客觀問題**，可在無逐像素驗證下正確推理，今晚即做。
- **視覺風格大改（階段 C）是主觀**，會以研究與分析為據、做可逆的小步 PR 並保留截圖供晨間
  review，不在使用者睡眠時擅自定案一個可能不合其品味的激進風格。
